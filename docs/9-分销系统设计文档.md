# 分销系统设计文档

> 模块：`aheadshop-distribution`（端口 8089）
> 数据库：`aheadshop_distribution`

---

## 一、整体架构

### 1.1 模块定位

新建独立微服务 `aheadshop-distribution`，遵循现有架构模式：

- 独立数据库 `aheadshop_distribution`
- 依赖 `common-core`、`common-mybatis`、`common-redis`、`common-exception`、`common-swagger`
- 通过 OpenFeign 调用 `user-service`、`order-service`
- 通过 RabbitMQ 监听订单支付/退款事件
- 通过 `aheadshop-admin` 聚合管理端接口

### 1.2 核心业务流程

```
用户A注册 → 系统自动生成邀请码 → 用户A分享邀请码/链接
    ↓
用户B注册时填写邀请码 → 绑定推荐关系（referrer_id = A的ID）
    ↓
用户B下单并支付成功 → order-service 发送 MQ 消息
    ↓
distribution-service 消费消息 → 计算佣金 → 写入佣金记录（冻结状态）
    ↓
订单完成超过7天 → 定时任务解冻佣金 → 可提现余额增加
    ↓
用户A申请提现 → 管理员审核 → 财务打款
```

## 二、数据库设计

```sql

    ADD COLUMN `referrer_id`  BIGINT     DEFAULT NULL COMMENT '推荐人ID（即邀请人的用户ID）',
    ADD UNIQUE INDEX `uk_invite_code` (`invite_code`),
```

| 字段 | 类型 | 说明 |
|------|------|------|
| invite_code | VARCHAR(16) | 注册后自动生成，全局唯一 |
| referrer_id | BIGINT | 注册时填写邀请码后写入，不可更改 |

#### `order` 表新增字段（aheadshop_order 库）

```sql

    ADD COLUMN `referrer_id` BIGINT DEFAULT NULL COMMENT '推荐人ID（用于佣金归属）',
    ADD INDEX `idx_referrer_id` (`referrer_id`);
```

## 三、核心业务逻辑

### 3.1 邀请码生成规则

- 格式：`AS` + 6位随机大写字母+数字（如 `AS3K7M2X`）
- 生成时机：用户注册成功后自动生成
- 唯一性：数据库唯一索引保证，冲突时重新生成
- 存储：写入 `user.invite_code` 和 `distributor.invite_code`

```java
public String generateInviteCode() {
    String chars = "ABCDEFGHJKLMNPQRSTUVWXYZ23456789"; // 去除易混淆字符 I/O/0/1
    String code;
    do {
        StringBuilder sb = new StringBuilder("AS");
        for (int i = 0; i < 6; i++) {
            sb.append(chars.charAt(ThreadLocalRandom.current().nextInt(chars.length())));
        }
        code = sb.toString();
    } while (existsByInviteCode(code)); // 唯一性检查
    return code;
}
```

### 3.2 注册绑定推荐关系

**触发时机**：用户注册接口 `/api/user/register` 新增可选参数 `inviteCode`

**处理流程**：

```
1. 校验邀请码格式（AS开头，共8位）
2. 查询邀请码对应的用户（user.invite_code = ?）
3. 校验：
   - 邀请码存在且用户状态正常
   - 不能自邀请（inviteCode 不是自己的）
   - 该用户尚未被推荐过（referrer_id 为空）
4. 设置 user.referrer_id = 邀请人的用户ID
5. 写入 referral_relation 表
6. 更新邀请人的 distributor.total_team_count + 1
```

**调用链路**：

```
user-service (register)
  → Feign: POST /inner/distribution/bind-referral
      { userId, inviteCode, referrerUserId }
  → distribution-service 写入 referral_relation
```

### 3.3 佣金计算

**触发时机**：订单支付成功后，order-service 通过 RabbitMQ 发送消息

**消息定义**：

```java
// Exchange: order.exchange
// RoutingKey: order.paid
// Queue: distribution.order.paid
public class OrderPaidMessage {
    private String orderNo;
    private Long userId;      // 买家ID
    private BigDecimal payAmount; // 实付金额
}
```

**处理流程**：

```
1. 查询 buyer 的 referrer_id（通过 user-service Feign）
2. 如果 referrer_id 为空 → 结束（无推荐人）
3. 查询 referrer 的 distributor 记录
   - 不存在或 status=禁用 → 结束
4. 计算佣金：
   - commissionAmount = payAmount × commissionRate
   - 默认佣金比例 10%（commissionRate = 0.1000）
   - 佣金金额向下取整到分
5. 写入 commission_record（status=冻结）
6. 更新 distributor：
   - frozen_commission += commissionAmount
   - total_commission += commissionAmount
7. 幂等处理：uk_order_no 唯一索引防止重复计算
```

### 3.4 佣金解冻（定时任务）

```java
@Scheduled(cron = "0 0 2 * * ?") // 每天凌晨2点
public void settleCommission() {
    // 查询冻结到期的佣金：
    // - status = 冻结(0)
    // - 对应订单状态 = 已完成(4)
    // - 订单完成时间超过 7 天
    List<CommissionRecord> records = findSettleable();

    for (CommissionRecord record : records) {
        // 1. 更新佣金状态：冻结 → 已结算
        record.setStatus(1);
        record.setSettleTime(LocalDateTime.now());
        commissionRecordMapper.updateById(record);

        // 2. 更新分销商余额
        Distributor distributor = distributorMapper.selectById(record.getDistributorId());
        distributor.setFrozenCommission(
            distributor.getFrozenCommission().subtract(record.getCommissionAmount()));
        distributor.setAvailableCommission(
            distributor.getAvailableCommission().add(record.getCommissionAmount()));
        distributorMapper.updateById(distributor);
    }
}
```

### 3.5 退款冲销

**触发时机**：订单退款成功后，order-service 发送 MQ 消息

**消息定义**：

```java
// Exchange: order.exchange
// RoutingKey: order.refunded
// Queue: distribution.order.refunded
public class OrderRefundedMessage {
    private String orderNo;
}
```

**处理流程**：

```
1. 查询 commission_record WHERE order_no = ?
2. 如果不存在或 status=已取消 → 结束
3. 更新佣金状态 → 已取消(2)
4. 更新分销商余额：
   - 如果原状态=冻结：frozen_commission -= amount
   - 如果原状态=已结算：available_commission -= amount
   - total_commission -= amount
```

### 3.6 提现流程

```
分销商申请提现：
  1. 校验 available_commission >= 提现金额（最低100元）
  2. 创建 withdrawal_record（status=待审核）
  3. 冻结对应金额（available → frozen，防止重复提现）

管理员审核通过：
  1. status → 已通过
  2. 财务线下打款后，标记 status → 已打款
  3. 扣减 frozen_commission

管理员拒绝：
  1. status → 已拒绝
  2. 金额退回 available_commission
```

## 十、安全与风控

### 10.1 防刷机制

| 风险 | 措施 |
|------|------|
| 邀请码暴力破解 | 邀请码8位+34字符集，组合数 34^6 ≈ 15亿，难以穷举 |
| 刷佣金（自买自） | 校验 buyer_id ≠ referrer_id |
| 重复计算佣金 | commission_record.order_no 唯一索引 |
| 重复提现 | 申请时立即冻结金额，防止并发提现超出余额 |
| 退款后佣金未冲销 | 监听退款 MQ 消息，同步冲销 |

### 10.2 数据一致性

- 佣金计算使用数据库事务保证 atomicity
- 分销商余额更新使用乐观锁（BaseEntity.version）
- MQ 消费端做幂等处理（订单号唯一索引）

```java
@Override @Transactional
public void settleCommission() {
    // 查冻结且创建时间超过 freezeDays(7天) 的佣金
    List<CommissionRecord> records = this.list(new LambdaQueryWrapper<CommissionRecord>()
        .eq(CommissionRecord::getStatus, CommissionStatus.FROZEN)
        .le(CommissionRecord::getCreateTime, LocalDateTime.now().minusDays(freezeDays)));
    for (CommissionRecord record : records) {
        // frozen → available
        Distributor d = distributionService.getByUserId(record.getDistributorId());
        d.setFrozenCommission(d.getFrozenCommission().subtract(record.getCommissionAmount()));
        d.setAvailableCommission(d.getAvailableCommission().add(record.getCommissionAmount()));
        distributionService.updateById(d);
        record.setStatus(CommissionStatus.SETTLED);
        record.setSettleTime(LocalDateTime.now());
        this.updateById(record);
    }
}
```
